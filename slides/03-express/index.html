No Cap. 2 sentimos a dor do roteamento manual com http nativo. O Express vem resolver.

Lembra da dor?

No módulo http nativo, cada rota era um if, sem middlewares, com o HTTP na mão.

if (req.method === "GET" && req.url === "/health") { ... }
if (req.method === "GET" && req.url.startsWith("/student")) { ... }
// ... e o código só cresce
O Express troca essa bagunça por peças bem definidas: rotas, middlewares, controllers.
Retomar o Cap. 2. Express é minimalista (unopinionated): você compõe.

Uma rota

Associação entre um verbo HTTP, um caminho e uma função handler.

import express from "express";
const app = express();

app.get("/usuarios", (req, res) => {
  res.json({ mensagem: "..." });
});

app.listen(3000);
req e res são enriquecidos pelo Express (res.json, req.params…). Rotas são avaliadas na ordem de declaração.

Verbos e semântica REST

Respeitar a semântica torna a API previsível para quem a consome.
201 Created e 204 No Content fogem do 200 padrão — por isso explícitos.

Três canais de entrada

CanalDe ondePara quê

params e query chegam sempre como string — converta com Number(...) quando precisar.
req.body exige o middleware express.json() (próximo bloco). Lembrar da conversão de tipo.

Router: dividir para organizar

// usuarios.routes.js
import { Router } from "express";
const router = Router();

router.get("/", listar);
router.get("/:id", obter);

export default router;
// app.js
app.use("/usuarios", router);

// router.get("/")    → GET /usuarios
// router.get("/:id") → GET /usuarios/:id
Cada recurso em seu módulo; o prefixo é herdado no app.use.
Router = mini-app: aceita rotas e middlewares próprios.

Middleware: o coração do Express

(req, res, next) => {
  // faz algo com a requisição...
  next(); // ...e passa o controle adiante
}
Se um middleware não chamar next() nem responder, a requisição trava (timeout no cliente).
Toda app Express é uma sequência de middlewares, executada de cima para baixo.

O pipeline

log (morgan)→ express.json()→ rota /saude→ res.json()

A requisição atravessa a cadeia até alguém responder — ou um erro ser lançado.

Cada um chama next() até a rota final enviar a resposta.

Quatro tipos

  • De aplicação — app.use(...), para todas (ou um prefixo).
  • De rota — presos a um Router.
  • De erro — assinatura de 4 args: (err, req, res, next).
  • De terceiros — pacotes npm da comunidade.
O Express reconhece o middleware de erro pela assinatura de 4 parâmetros.
next(err) pula direto para o middleware de erro.

Middlewares de terceiros essenciais

helmetcabeçalhos de segurança
corsquais origens podem consumir a API
morganlog das requisições
express-rate-limitlimita requisições por IP (429)
multerupload de arquivos
compressioncomprime a resposta (gzip)
A ordem importa: helmet/cors primeiro, erros por último.

A ordem importa (app.js)

app.use(helmet());                 // 1. segurança
app.use(cors());                   // 2. origens
app.use(compression());            // 3. performance
app.use(morgan("dev"));            // 4. logging
app.use(express.json());           // 5. parsing do corpo
app.use("/api", router);           // 6. rotas
app.use(middlewareDeErros);        // 7. erros — sempre por último
Mnemônico: segurança → performance → log → parsing → rotas → erros.

Erros assíncronos: cuidado

❌ exceção some:

app.get("/u", async (req, res) => {
  const u = await buscar(); // pode falhar
  res.json(u);
});

✅ delega ao next:

app.get("/u", async (req, res, next) => {
  try { res.json(await buscar()); }
  catch (err) { next(err); }
});
Um asyncHandler(fn) aplica o try/catch automaticamente.
Em rotas async, exceção não capturada NÃO chega ao middleware de erro sozinha.

Controllers: separar papéis

Rota→ Controller→ Service

O controller só coordena: lê req, chama o service, responde com res.

Controller não tem regra de negócio nem acesso ao banco — isso é do service.
Separation of Concerns. Rotas = quais; controller = como; service = o quê.

Controller enxuto + rota declarativa

// controller
export const obterUsuario =
  async (req, res, next) => {
  try {
    const u = await service
      .buscarPorId(Number(req.params.id));
    if (!u) return res.status(404)
      .json({ erro: "não encontrado" });
    res.json(u);
  } catch (e) { next(e); }
};
// routes
router.get("/", listarUsuarios);
router.get("/:id", obterUsuario);
router.post("/",
  validar, criarUsuario);
router.put("/:id", atualizarUsuario);
router.delete("/:id", removerUsuario);
As rotas ficam declarativas; a lógica vive nos controllers.
O service usa array em memória por ora; troca por ORM depois sem mexer no controller.

Estrutura de projeto

src/
├── config/        env, banco
├── controllers/   coordenam requisições
├── middlewares/   auth, validação, erros
├── routes/        index.js + um por recurso
├── services/      regra de negócio
├── utils/         AppError, asyncHandler
└── app.js         configura o Express
server.js          sobe o servidor
Organização por tipo de arquivo (comum em apps pequenas/médias).

app.js ≠ server.js

// app.js
const app = express();
app.use(/* middlewares */);
app.use("/api", router);
export default app;
// server.js
import app from "./src/app.js";
app.listen(PORTA);
Separar permite testar o app (supertest) sem subir a porta.
Amarra com o Cap. 9 (testes). server.js só o entry point.

Erro padronizado

export class AppError extends Error {
  constructor(mensagem, statusCode = 500) {
    super(mensagem);
    this.statusCode = statusCode;
  }
}
// throw new AppError("Usuário não encontrado", 404);
O middleware de erro lê statusCode e monta a resposta — sem if espalhado pelo código.
Uma classe de erro com status → respostas de erro consistentes.

Resumo

  • Rota = verbo + caminho + handler; dados via params/query/body.
  • Middleware = (req, res, next); a app é um pipeline.
  • Terceiros: helmet, cors, morgan… — a ordem importa.
  • Controller coordena; o service tem a regra de negócio.
  • app.js configura, server.js sobe — testável.
