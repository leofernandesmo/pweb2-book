Já entendemos a Web e o HTTP (Cap. 1). Agora: com o que vamos construir o servidor? Node.js. Antes do Express, entender o runtime por baixo.

Um garçom, muitas mesas

Um bom garçom não fica parado ao lado da mesa esperando a cozinha ficar pronta.

Ele anota o pedido, atende outras mesas e volta quando a cozinha avisa.

É exatamente assim que o Node.js lida com muitas requisições ao mesmo tempo: sem travar à espera de I/O.
Analogia do event loop. Cozinha = banco de dados / serviço externo (I/O lento). O garçom (a thread) não bloqueia esperando.

Onde queremos chegar

  • Entender o Node.js como runtime orientado a eventos e I/O não bloqueante.
  • Gerenciar um projeto com npm: package.json, dependências e scripts.
  • Organizar código em módulos (ESM: import/export).
  • Construir um servidor HTTP com o módulo nativo — e sentir sua dor.
  • Testar a API pelo terminal com cURL.

O que é o Node.js?

Um ambiente de execução de JavaScript fora do navegador, sobre o motor V8 (o mesmo do Chrome).

Modelo orientado a eventos
I/O não bloqueante
Single-thread + event loop

Ideal quando o gargalo é esperar (banco, rede, arquivos) — o caso típico de uma API REST.

V8 compila JS para código nativo. Node adiciona APIs de sistema (fs, http, net…) e o event loop (libuv).

Bloqueante × não bloqueante

❌ Bloqueante — trava a thread:

const d = lerSync("aluno.json");
console.log("segue");
// nada roda até o arquivo chegar

✅ Não bloqueante — libera a thread:

const d = await ler("aluno.json");
console.log("segue");
// o Node segue atendendo
Uma thread só, mas muitas requisições em voo — porque ninguém fica parado esperando I/O.
Reforçar: não-bloqueante não é "mais rápido para uma tarefa", é "não desperdiça a thread esperando". Concorrência, não paralelismo.

De onde veio

  • 2009 — Ryan Dahl cria o Node para escalar sob alta concorrência sem uma thread por conexão.
  • npm consolida um ecossistema gigante de bibliotecas.
  • Hoje: governança da OpenJS Foundation, versões LTS, async/await e ES Modules.
Há um documentário ("Node.js: The Documentary") — bom para passar como material extra.

npm: o projeto em um arquivo

npm init -y      # cria o package.json com padrões
{
  "name": "api-academica",
  "type": "module",
  "scripts": { "start": "node src/server.js" },
  "dependencies": { }
}
"type": "module" ativa os ES Modules (import/export).
package.json = manifesto do projeto: nome, dependências, scripts, tipo de módulo.

Dependências

npm install express     # produção  (dependencies)
npm i -D jest           # desenvolvimento (devDependencies)

Instalar gera três efeitos:

entrada no package.json+ node_modules/+ package-lock.json
Só as de produção vão para o deploy; as de dev (testes, linters) ficam de fora.
package-lock trava as versões exatas — builds reprodutíveis. node_modules nunca vai para o Git (.gitignore).

Scripts padronizam a equipe

"scripts": {
  "start": "node src/server.js",
  "dev":   "node --watch src/server.js",
  "test":  "jest"
}
npm run dev     # ou: npm start / npm test
Todo mundo roda o projeto do mesmo jeito — sem "na minha máquina funciona".
--watch (Node 18+) reinicia ao salvar — sem nodemon.

Modularização

Uma app real não cabe num arquivo só. Separe por responsabilidade:

src/
├── server.js
├── routes/       studentRoutes.js
├── controllers/  studentController.js
└── services/     studentService.js
Já é o embrião do MVC — formalizado no Cap. 4.
Rota → Controller → Service: cada camada com um papel.

export → import (ESM)

// studentService.js
export function findStudentById(id) {
  return { id, name: "Maria" };
}
// studentController.js
import { findStudentById }
  from "../services/studentService.js";

export function getStudent(req, res) {
  res.json(findStudentById(req.params.id));
}
Cada módulo expõe o que outros podem usar — e nada mais.
Em ESM, o caminho relativo precisa da extensão .js. Nomeado vs default export.

Servidor HTTP nativo

Antes do Express, veja o que ele abstrai. O módulo http puro:

import http from "http";

const server = http.createServer((req, res) => {
  res.statusCode = 200;
  res.setHeader("Content-Type", "text/html");
  res.end("<h1>Olá, Mundo!</h1>");
});

server.listen(3000, () => console.log("http://localhost:3000"));
createServer recebe um callback (req, res) por requisição. req = pedido, res = resposta. res.end() fecha o fluxo.

Roteamento... na mão

const server = http.createServer((req, res) => {
  if (req.method === "GET" && req.url === "/health") {
    res.writeHead(200, { "Content-Type": "application/json" });
    return res.end(JSON.stringify({ status: "ok" }));
  }
  if (req.method === "GET" && req.url.startsWith("/student")) {
    res.writeHead(200, { "Content-Type": "application/json" });
    return res.end(JSON.stringify({ id: 1, name: "João" }));
  }
  res.writeHead(404);
  res.end();
});
Cada rota é um if. Método + URL checados à mão. Repare como cresce.

A dor que motiva o Express

Com o módulo nativo, tudo é manual:

  • roteamento é uma cadeia de if;
  • não há middlewares nem parsing de corpo pronto;
  • o tratamento de erros é espalhado.
A cada rota nova, o código fica menos coeso. É isso que o Express (Cap. 3) vem resolver.
Não é que o módulo nativo seja ruim — entender ele evita tratar o Express como caixa-preta.

Testando com cURL

curl http://localhost:3000/health
# {"status":"ok"}

curl -X POST http://localhost:3000/student \
  -H "Content-Type: application/json" \
  -d '{"name":"Maria"}'

curl -i http://localhost:3000/health   # -i mostra os cabeçalhos
O terminal é seu cliente HTTP: método, corpo, cabeçalhos e status, tudo à vista.
-X método, -H cabeçalho, -d corpo, -i inclui headers. Amarra com o HTTP do Cap. 1.

Resumo

  • Node = runtime de JS, orientado a eventos e não bloqueante.
  • npm gerencia dependências e scripts via package.json.
  • Módulos ESM separam responsabilidades (embrião do MVC).
  • O HTTP nativo funciona, mas vira bagunça — daí o Express.
  • cURL testa a API direto do terminal.
