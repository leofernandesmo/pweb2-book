No Cap. 3 a API já funciona. Mas funcionar não é ter boa arquitetura. Aqui separamos camadas.

O controller gordo

export const criarUsuario = async (req, res) => {
  if (!email.includes("@")) return res.status(400)...   // validação
  const existe = await db.query("SELECT ...");           // banco
  const hash = await bcrypt.hash(senha, 10);             // regra
  await db.query("INSERT ...");                          // banco
  res.status(201).json(...);                             // resposta
};
Validação + regra de negócio + SQL + resposta, tudo numa função só.
Fat controller: o antipadrão. Deixe a turma apontar os problemas.

Por que isso dói?

  • Impossível de testar sem um banco real.
  • Não dá para reutilizar (ex.: um job em background).
  • Toda mudança de regra mexe no controller.
A cura: distribuir responsabilidades entre camadas.
Isso motiva todo o capítulo.

O caminho

MVC em REST→ Service→ Repository→ Injeção de Dependência

Quatro peças que transformam o controller gordo em uma base testável e fácil de evoluir.

Fecha com a refatoração do projeto do Cap. 3.

MVC em APIs REST

CamadaNa API REST
Modelestrutura do recurso + acesso a dados
Viewa resposta res.json() (não há HTML)
Controllercoordena; nunca faz regra de negócio
Sem interface visual, a View vira JSON — o resto do padrão continua valendo.
MVC nasceu no Xerox PARC (Smalltalk, anos 70). Adaptado: View = serialização JSON.

Service: a guardiã da regra

Expressa o que a aplicação faz — sem saber como os dados chegam nem onde ficam.

Regra de ouro: o service não conhece req nem res.
// ❌ acoplado ao Express
criar(req) {
  const { nome } = req.body;
}
// ✅ dados puros
criar({ nome, email, senha }) {
  // reutilizável em qualquer contexto
}
Se o service toca req/res, está preso ao HTTP. Controller extrai do req e monta o res.

Duas validações diferentes

Entrada

formato de e-mail, tamanho de senha, campo obrigatório.

→ middleware de validação (contrato da API)

Domínio

"e-mail não pode repetir", "produto sem estoque não vende".

→ service (regra de negócio)
Distinção importante: entrada = forma; domínio = regra. Lugares diferentes.

Repository: onde os dados vivem

Uma camada entre o service e a persistência, com interface orientada ao domínio: buscarPorId, criar, listarTodos.

Service→ contrato → Repository→ memória / SQL / ORM
O repositório não tem regra de negócio — só persiste, recupera e consulta.
O service não sabe se é Postgres, Mongo ou memória. Só chama métodos do contrato.

Mesmo contrato, implementações diferentes

// em memória (agora)
class UsuariosRepository {
  async buscarPorId(id) {
    return this.usuarios
      .find(u => u.id === id) ?? null;
  }
}
// Prisma (Cap. 5)
class UsuariosRepositoryPrisma {
  async buscarPorId(id) {
    return prisma.usuario
      .findUnique({ where: { id } });
  }
}
Mesmos métodos → trocar a implementação não muda o service.
Essa identidade de interface é proposital. É o gancho para DI.

Inversão de Dependência

❌ cria a própria dependência:

class UsuariosService {
  constructor() {
    this.repo =
      new UsuariosRepository();
  }
}

✅ recebe pelo construtor:

class UsuariosService {
  constructor(repo) {
    this.repo = repo;
  }
}
Injetar a dependência desacopla: o service aceita qualquer repositório que cumpra o contrato.
new dentro da classe = acoplamento estático. Injeção via construtor resolve.

Composition root

// usuarios.routes.js — um único lugar monta as peças
const repository = new UsuariosRepository();
const service    = new UsuariosService(repository);
const controller = new UsuariosController(service);
As dependências são compostas uma vez, no ponto de entrada. Cada camada conhece só a interface da vizinha.
DI (técnica) implementa o DIP (princípio). Não são a mesma coisa.

O prêmio: testabilidade

it("lança 409 se e-mail já existir", async () => {
  const repoFalso = {
    buscarPorEmail: async () => ({ id: 1 }),
  };
  const service = new UsuariosService(repoFalso);
  await expect(service.criar({ email: "ana@x.com", ... }))
    .rejects.toThrow("E-mail já cadastrado");
});
Testa a regra de negócio sem banco — rápido, determinístico, isolado.
Injetar um mock no lugar do repositório real. Amarra com o Cap. 9.

Resumo

Controller→ Service→ Repository→ dados
  • Controller coordena; View = JSON.
  • Service tem a regra e não conhece req/res.
  • Repository abstrai onde os dados vivem (contrato).
  • Injeção desacopla e torna tudo testável.
